# 不要给 Postgres 太多内存

我时不时地会调查某种批处理的问题。此类进程使用非常高的内存限制（maintenance_work_mem和work_mem）变得越来越普遍。我想一些 DBA 遵循"越多越好"的逻辑，却没有意识到这会对性能造成很大的损害。

让我用我在测试 GIN 索引并行构建的 afix 时遇到的一个示例来演示这一点。这个错误不是特别有趣或复杂，但它需要相当高的maintenance_work_mem值（最初的报告使用了 20GB）。

为了验证修复，我对一系列maintenance_work_mem值和不同数量的并行工作程序运行了 CREATE INDEX。目标是检查其中一些是否仍然失败，但我也测量了时间。我得到了这张图表：

![](https://fastly.jsdelivr.net/gh/bucketio/img4@main/2025/11/01/1761998091696-260863a5-2907-4e4c-be42-8e83807a5eaa.png)

这来自 Azure 上的 D96v4 实例，在 6x NVMe 上具有 Xeon Platinum 8573C、384GBRAM 和 RAID0。这使得它完全缓存并受 CPU 限制。

并行度有很大帮助，2 个工作程序的速度要快 ~1.8 倍（领导者算作一个工作程序）。这几乎是完美的加速，因为构建的最后阶段仍然是串行的。加速随着更多的工人而降低（8 个工人，它只有 ~4.5 倍），但这是意料之中的。

但图表也显示，增加maintenance_work_mem会变慢！从 64MB 到 16GB 的持续时间增加了 30%，无论使用多少工作程序。

# 为什么会这样？

可能导致这种行为的因素可能有多种。让我解释一下我认为最重要的两个。

首先，是 L3 缓存大小。

系统中的所有 RAM 速度并不相同，有几层性能差异很大。一小部分 RAM 集成到 CPU 中，因此延迟非常低。大多数CPU只有~32-128MB的这种快速的CPU上RAM，但它比主内存快一个数量级。

在构建索引时，我们通常会将数据累积到一个缓冲区中，当缓冲区"已满"时进行处理，然后将其组合成最终索引。对于 GIN 索引，我们将条目添加到哈希表中，这意味着随机内存访问。当哈希表溢出 L3 时，它必须更频繁地访问主内存。这比访问 L3 贵得多，差异约为 20 个周期和 200 个周期。

通常最好以较小的块处理数据，其大小使其适合 L3 缓存。当然，我们需要更多的块，但这仍然可能是胜利。

> 注意：在一篇论文中有很多关于内存的有趣信息，每个[程序员都应该知道的关于内存](https://people.freebsd.org/~lstewart/articles/cpumemory.pdf)，作者是 Ulrich Drepper。这是 2007 年的事了，但基本面没有改变。

还有其他原因可以解释为什么处理数据可能更好更小块。较大的块可能会增加系统其他部分的压力，甚至导致失速。

例如，当哈希表超过内存限制（由maintenance_work_mem设置）时，数据将写入磁盘。这些是临时文件，没有持久性要求，因此仅将数据写入页面缓存是可以的。

然而，内核还试图使用两个阈值来控制脏数据的数量。点击vm.dirty_background_ratio后它开始在后台写出脏数据，而用户写仍然转到页面缓存。如果脏数据量不断增长并命中vm.dirty_ratio，则所有写入都将被迫同步。

理想情况下，后台写入足以控制脏数据量，并且永远不会达到vm.dirty_ratio阈值。这种主动刷新的工作效果取决于内核需要响应的时间。更频繁地以更小的块写入数据，让内核有更多时间来刷新一些数据。

假设您在哈希表中积累 8GB 的数据需要一分钟。您可以等待 8GB，然后一次性写出。大多数时候不会写入，然后会有很短的时间写入 8GB 的。或者您可以累积并写出 64MB 的块。在这种情况下，随着时间的推移，写入将更加分散，从而给内核做出反应的时间。

当然，在某些情况下，积累更多数据可能会更好。例如，它可以提高压缩比。很难说。

# 总结

当然，所有这些也适用于work_mem。唯一的区别是work_mem用于常规查询，而不是用于维护作。但推理是完全一样的。哈希连接构建一个哈希表，受work_mem限制。如果哈希表超过L3，它会变慢。对于哈希聚合、排序和各种其他作也是如此。

我不确定maintenance_work_mem和work_mem的"最佳"设置是什么。但这不是这篇文章的主题。我的观点是盲目设置一个非常高的值可能是相当有害的。

我的建议是坚持适度的值(例如 64MB)，并且只有在可以证明好处的情况下才调整值。
