# 查询计划多久优化一次？

查询优化器的基本承诺是它选择"optimal"查询计划。但有一个问题 - 计划选择依赖于成本估算，根据选择性估算和基本资源（I/O、CPU 等）的成本计算得出。所以问题是，我们多久选择一次"fastest"的计划？事实是，我们实际上经常犯错误。

考虑以下图表，其中包含带有范围条件的简单 SELECT 查询的持续时间。条件会发生变化以匹配表格的不同部分，显示在 x 轴上（具有匹配行的页面的分数）。计划被迫使用enable_选项使用不同的扫描方法，并且即使不使用enable_参数，当扫描方法“获胜”时，暗点标记也会运行。

![](https://fastly.jsdelivr.net/gh/bucketio/img16@main/2025/11/01/1762000944196-c3b020c7-219c-431e-a4c3-f8d60c60abfe.png)

它表明，对于选择性 ~1-5%（x 轴是对数），规划器选择索引扫描，但这恰好是一个糟糕的选择。它最多需要 ~10 秒，一个简单的“哑”顺序扫描将在 ~2 秒内完成查询。

这很好地证明了为什么尝试到处获取索引扫描是毫无意义的，而应用程序开发人员有时会以此为目标。

1% 的选择性相当高，但对于较低的选择性，它没有太大区别。它在图表上不是很明显，但让我们将 they-axis 切换到对数刻度，以便更好地查看低选择性：

![](https://fastly.jsdelivr.net/gh/bucketio/img9@main/2025/11/01/1762000970930-272a3702-7953-4063-a776-632d685b2feb.png)

索引扫描永远不会赢得此查询！位图扫描占主导地位，即使对于最具选择性的条件，也高出 ~10 倍。规划器确实正确地选择了位图扫描，所以这很好。

**注意**：原因很简单 - 位图扫描执行预取，索引扫描不执行。禁用预取（或使用我的 WIP 补丁和索引预取）可以消除差异。

上面的图表是针对统一数据集的，这有点极端。一方面，它是计算估计最简单的。我们甚至假设在某些地方是统一的/独立的。另一方面，它的数据局部性最差。

那么，其他一些具有不同相关模式的数据分布呢？我用许多数据集运行了这个，这里有几个例子。我将显示每个的常规和对数比例表。

# cyclic, fuzz=1

首先，"cyclic"，带有一点"fuzz"，增加了随机性。

![](https://fastly.jsdelivr.net/gh/bucketio/img1@main/2025/11/01/1762001125662-be354757-ba90-4c8f-a143-fff330504b16.png)

![](https://fastly.jsdelivr.net/gh/bucketio/img3@main/2025/11/01/1762001138404-fb50a279-e40a-4e3d-bdd4-98eb54748cb2.png)

这（也许）比统一数据集好一点，但我们仍然犯了一些错误。我们暂时切换到索引扫描，而不是仅仅坚持位图扫描。然后我们过早地转向 seqscan。

# linear, fuzz=10

首先，"linear"数据集，"fuzz"增加了 10 倍的随机性。

![](https://fastly.jsdelivr.net/gh/bucketio/img15@main/2025/11/01/1762001182281-00c671e5-6608-410a-932d-1689cae45bc6.png)

![](https://fastly.jsdelivr.net/gh/bucketio/img1@main/2025/11/01/1762001195402-a0b540a0-8af4-4a8c-a48a-c6c1debf2a64.png)

不是很好，而且显然比我们在统一图表上看到的更糟糕。规划器在选择索引扫描方面更加持久，即使对于低选择性（在统一上它选择位图扫描）。这些决策都不是最佳的 - 位图扫描将是更好的选择。只有在最后（选择性~90%），计划才能正确翻转到seqscan。

# linear (no fuzz)

最后，一个完美的线性数据集 - 单调递增的值。这恰好是我们始终选择正确（实际上是最快的）计划的唯一情况。

![](https://fastly.jsdelivr.net/gh/bucketio/img2@main/2025/11/01/1762001224269-1198676a-499f-4048-aca5-f7f43c3854d2.png)

![](https://fastly.jsdelivr.net/gh/bucketio/img0@main/2025/11/01/1762001240279-00d1c748-33b8-4ecb-951b-fd61b2fb08a5.png)

索引扫描和位图扫描之间几乎没有区别。该数据集非常适合内核预读，并且不会受益于手动预取。翻转到 seqscan 也在正确的位置（即使它没有翻转，索引扫描也可以正常工作）。

# 总结

所呈现的结果来自配备 Ryzen 9900X 和 NVMe RAID 的机器。我从其他硬件（例如 SATA SSD）获得的结果要多得多，但整体行为完全相同。

运行是使用冷缓存（页面缓存或共享缓冲区中没有任何内容）。使用暖缓存，一些差异会消失，或者至少变得不那么重要。例如，indexscans 和位图扫描之间的所有差异都消失了，因为预取对缓存中的所有数据几乎没有影响。

我试图表达的观点是，规划者根据统计数据和成本模型选择它认为是最佳的计划。如果这真的是最好的计划，那就太好了，但它可能不是，而且往往不是。我们所能希望的最好的事情就是不要离最好的计划太远。

这可能不会立即显现出来，因为你怎么知道有更好的计划？然后你必须手动强制执行计划。大多数用户只是信任数据库，并且只注意到这一点在不同计划之间的阈值处的选择性。这就是“悬崖”发生的地方，当查询在性能截然不同的不同计划之间切换时。

也许可以调整一些成本参数以获得更好的计划更频繁。我的经验是，对于具有一种分布的数据集有效的方法可能会损害另一个数据集。所有数据集的完美配置可能是不可能的。

此外，测试使用非常简单的查询 - 具有单个范围 WHERE 条件的单表 SELECT。对于更复杂的查询（例如，具有不同数据分布的表的连接），行为将更加复杂。

有没有一个合理的方法来改善这一点？规划必须依赖于非常紧凑/小的数据表示（优化统计数据），以便快速。而且统计数据必然只是数据的简化表示。我们可以改进这一点并添加更多统计数据（例如，关于列之间的相关性等），但它始终是无损压缩。它总是会省略一些有趣的细节。

还有成本模型本身的问题，这是硬件行为的非常粗略的近似值。在本地系统中很粗糙，在具有复杂存储系统等的云环境中似乎更不准确。

这似乎是基于成本的规划的一个相当根本的挑战。我确信我们可以改进各个步骤（更好的统计数据、更好的成本模型），但总会有一些缺失的细节。

话虽如此，我仍然认为基于成本的规划是最好的可用方法。因为有什么更好的选择呢？当然不是硬编码的计划，也不是基于规则的计划。但它提醒了我计划稳健性的重要性，甚至可能有办法调整查询计划，如果我们意识到它不太正确。
