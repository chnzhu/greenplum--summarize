# 为什么不选择最佳查询计划呢？

## 重现结果

我分享了我用来生成数据集和查询的[SQL](https://vondra.me/posts/so-why-dont-we-pick-the-optimal-query-plan/create.sql>)和[shel](https://vondra.me/posts/so-why-dont-we-pick-the-optimal-query-plan/run-test.sh
)l脚本，但我忘记提及几个重要的细节。不幸的是，这影响了可重复性。

### SQL 脚本（数据生成器）

SQL 脚本需要两个参数：

- ROWS - 要插入到表格中的行数
- CYCLES - 循环数据集中的"cycles"数

对于我的测试，我使用了 10M 行和 100 个周期，这使得每个表大约 4.3GB。像这样的东西应该会给你大约相同的数据库。

```sql
$ sed 's/ROWS/10000000/g' create.sql | sed 's/CYCLES/100/g' | psql testdb
```

脚本生成随机数据，但大小和数据分布/模式应相同。

### bash 脚本（查询生成器）

bash 脚本需要三个位置参数：

- dbname - 要运行的数据库（所有数据都已使用 SQL 脚本预生成）
- datadir - Postgres 数据目录（以便可以重新启动）
- timeout - 查询的超时（这只是通常的 statementTimeout（以毫秒为单位）

所以这样运行它：

```sql
$ ./run-test.sh testdb /mnt/pg/data-directory 60000
```

您可能需要稍微调整超时，具体取决于您使用的存储类型。目标是允许大多数查询完成，并且仅中止一些极端情况。

### Postgres 配置

最后一部分是实例的配置。我只做了最基本的参数调整，特别是：

- shared_buffers = 32GB
- random_page_cost = 2

shared_buffers值对执行有一点影响，但在规划过程中不考虑它。random_page_cost参数非常重要，它以一种有趣的方式影响规划。我稍后会详细讨论它。

所有其他参数均保留为默认值。

有了这个，你应该能够重现我在我的帖子中描述的相同行为。您还需要一个简单的[drop-caches.sh](https://vondra.me/posts/so-why-dont-we-pick-the-optimal-query-plan/drop-caches.sh
)脚本，它只是从页面缓存中逐出所有内容以获得“冷”查询持续时间。

当然，确切的行为取决于硬件，但如果您在某种闪存（最好是本地 SSD）上重新运行，您应该看到我介绍的相同行为。

### random_page_cost

我花了一段时间才弄清楚为什么有几个人努力重现我看到的糟糕的计划选择。然后我意识到它们正在使用默认参数运行，尤其是 random_page_cost = 4。一旦他们将其更改为较低的值 2，他们就会得到相同的“索引扫描”计划。为什么？

考虑下图，其中包含random_page_cost值为 1.25、2 和 4 的三种基本扫描类型的估计成本。这是针对统一数据集的，大约是最简单的数据分布类型。使用PNG 或 PDF 获取图表的更高分辨率版本。

![](https://fastly.jsdelivr.net/gh/bucketio/img10@main/2025/11/01/1761999574020-da1d7832-8db8-468d-9d4d-0f71f7c9cfd9.png)

random_page_cost值对顺序扫描（蓝点）没有影响，位图扫描似乎也几乎不受影响（成本确实发生变化，但变化不大）。

索引扫描的成本变化很大。这是完全有道理的，因为索引扫描会执行大量随机 I/O，这是成本的主要部分。random_page_cost越低，索引扫描越便宜。它在 10% 到 100% 之间的选择性上清晰可见。

总成本与random_page_cost值大致成正比。从 4 到 2 将成本降低到 50%，而从 1.25 意味着大约 30%。至少对于 10-100% 的选择性，接近 0% 时不太明显。

但规划者并不关心个人计划隔离的成本，它关心它与其他计划的比较。这些图表显示，随random_page_cost值的降低，索引扫描开始更频繁地胜出（胜过位图扫描）。暗点显示优化器选择了哪个扫描。红色的数量增加，这意味着规划者选择指数扫描作为最便宜的。

### 冷数据

以下是相同random_page_cost值（PNG 或PDF）的"冷运行"（未缓存数据）的持续时间图表。

![](https://fastly.jsdelivr.net/gh/bucketio/img5@main/2025/11/01/1761999632222-6b8bc683-14bb-4ef6-bf1f-69791a457611.png)

对于不同的random_page_cost值，形状根本不会改变，这是有道理的 - 参数只对规划很重要。一旦选择了一个计划，该值就变得无关紧要了。计划的持续时间不取决于它的选择方式。

但是，选择指数扫描作为"最佳"计划的转变更常见仍然适用。random_page_cost越低，选择性就越高，规划者认为这是最便宜的计划。对于random_page_cost = 4.0，索引扫描仅在 ~1% 时获胜一次。对于random_page_cost = 1.25，它在 1% 到 10% 之间的任何地方都获胜。在所有这些情况下，它比位图扫描慢得多（甚至与 seqscan 相比）。

上图是针对冷数据的持续时间，即 sharedbuffers 和页面缓存都是空的。如果数据被缓存，您将获得此图表（高分辨率 PNG 或 PDF）。

![](https://fastly.jsdelivr.net/gh/bucketio/img14@main/2025/11/01/1761999690205-22dd167f-a018-40a9-b08a-31d1f987f2be.png)

我们看到计划之间的相同变化，但如果没有 I/O，差异大多可以忽略不计。它很好，很流畅，而且不会慢多少比"dumb"顺序扫描。

### "random_page_cost = 2.0"错了吗？

那么我在换random_page_cost的时候是不是调音错误了呢？我真的不这么认为。默认值 （4） 此时已有数十年的历史，当每个人都使用"old"旋转盘时，它被重置了。虽然即使在现代存储（如 NVMe 闪存）上，随机 I/O 和顺序 I/O 之间仍然存在差异，但它正在变得越来越小。基于此，降低价值是有意义的。

我知道建议将 random_page_cost 设置为更接近 1.0，甚至官方文档也提到可能将 SSD 设置为 1.1：

> 相对于顺序随机读取成本较低的存储，例如固态驱动器，也可以更好地使用较低的random_page_cost值进行建模，例如 1.1。

这可能太激进了。即使使用 NVMe SSD，随机 I/O 仍然比顺序 I/O 更昂贵。因此，除非您假设所有内容都已缓存（如上面的“温暖”情况），否则我会坚持使用更高的值。也许如果你对你的数据集和工作负载进行一些仔细的测试，但是没有那个 2.0 似乎是一个合理的折衷方案。

由于其他一些估计值大幅偏离，估计成本也可能偏离。对于索引扫描，成本是通过cost_index() 计算的，所以它可能错误地估计了一些输入参数？比如说，要访问的堆数？我确实检查了所提供的示例，但我认为这种情况不会发生。这并不奇怪，均匀分布是所有公式的默认假设，相当容易估计。在这种情况下，估计往往是最准确的。

### 如何识别问题？

那么，您如何确定自己受到这种行为的影响呢？我不确定。

如果您在快速计划和慢速计划之间的"threshold"上执行查询，您可能会注意到有两种模式。这就是我的"performance cliffs"演讲中讨论的情况。这很容易注意到。

但是，如果您的查询始终属于"slow"区域，则只会看到缓慢的执行。你怎么知道一个更快的计划是可能的？你必须猜测有一个更快的计划，然后尝试enable_选项，直到你说服计划者选择它。

这样的实验只能对少数相对简单的查询进行。随着查询变得越来越复杂和/或查询数量增加，探索它们变得困难。你可能会关注昂贵的查询，但你仍然不知道是否存在更好的计划。

### 我们如何解决/改进这个问题？

我犹豫是否要将这种行为称为错误，因为它按设计工作。当然，我不能排除错误的可能性。随意调查，我可能错过了什么。

我们可以在数据库本身中对此做些什么吗？我有几个想法......

### 我们应该调整random_page_cost默认值吗？

在所提出的示例中，根本问题似乎是索引扫描的成本核算。也许这都是因为random_page_cost太低了？也许是时候重新评估默认值应该是什么了？或者也许我们应该有一些自动化的方法来调整特定硬件的参数？

时不时地有人建议使用一个类似于pg_test_fsync 的工具来执行一些 I/O 测试并计算"correct"的random_page_cost值。也许我们应该尝试实现这样的工具，但有一个问题。它可能不能只是一个"simple"的硬件测试，因为该值不是随机与顺序 I/O 成本的简单比率。

AFAIK 当前的默认值 （4.0） 是根据运行不同扫描类型的查询的基准选择的，而 4.0 恰好在“正确的地方”切换了计划。我确信它在pgsql-hackers 的一些旧线程中进行了描述，但我目前找不到它。

因此，选择了一个值，以便成本模型做出正确的决策，这有点倒退。

如果在工具中实现这一点太困难了，也许我们至少应该重做基准测试并相应地调整random_page_cost默认值（或调整建议）？

预计我们最终会得到更低的rantom_page_cost值，因为硬件在处理随机 I/O 方面变得更好。但我并不完全确定这一点。我们改进了 Postgres 的所有部分，其中一些可能使顺序扫描更便宜。作系统还改进了相关位（如预读等）。如果我们真的最终得到了更高的random_page_cost违约，我不会感到惊讶。

### 让成本模型意识到预取？

我想到的另一件事是，这可能是由于优化器没有考虑预取。位图扫描预取堆页，而索引扫描则不这样做。然而，优化器却没有意识到这一点。

当 effective_io_concurrency = 1 时，这可能并不那么明显，但 PG 18 将默认值增加到 16，这可能对位图扫描有很大帮助。也许是时候在进行 costingbitmap 扫描时考虑这一点了？

我们也可以做相反的事情，为索引扫描添加预取。我们需要这样做是出于其他原因（例如支持直接 I/O），并且我已经为此工作了很长一段时间。这是其中一项测试的图表，包括带预取的索引扫描：


![](https://fastly.jsdelivr.net/gh/bucketio/img18@main/2025/11/01/1761999974284-30c1ed13-b515-4bac-bca9-6117497caddc.png)

预取将使索引扫描与位图扫描相当，至少对于高达 ~10% 的"medium selectivity"。超过此阈值，位图扫描受益于仅访问每个页面一次（并且以顺序模式），而索引扫描必须重复访问它。

但即便如此，当扫描 100% 的表格时，预取也会使索引扫描速度提高 2 倍。

# 总结

最终，我们在优化器中能做的只有这么多。它根据数据集的非常简化的近似值做出决策，因此它必然会犯错误。而且我们并没有通过向执行器添加越来越多的选项来使其任务变得更简单。每个选项以不同的方式运行查询意味着犯错误的机会。

我认为不可能通过让计划者更聪明来解决这个问题。当然，我们应该不断改进计划器。还有一些有趣的想法，比如更好的统计数据，自适应优化，或者使用过去执行的反馈。我们应该探索这一点。

但这种先验的决定根本不可能 100% 正确。一个从不犯错误（对任意数据进行任意查询）的“完美”规划器是一个无法实现的目标。

我们需要改进执行者，使其更能抵御此类错误，并优雅地处理它们。有一个完整的研究学科专注于查询执行的稳健性。
